# Feedback Parser

Превращает сырую, неструктурированную обратную связь в чистый, единообразный формат для передачи в разработку.

## Когда использовать

Пользователь загружает данные (текст, таблицу, CSV, скопированные заметки) содержащие обратную связь — баги, предложения, жалобы, вопросы — и хочет получить структурированный результат.

## Процесс обработки

### 1. Определи источник и формат входных данных

Входные данные могут быть в любом виде:
- Таблица (TSV/CSV, Excel, Google Sheets копипаст)
- Свободный текст (заметки с встречи, переписка)
- Список пунктов
- Микс языков (русский, казахский, английский)
- Смесь нескольких отзывов в одном документе

Не проси пользователя переформатировать — работай с тем что есть.

### 2. Извлеки каждый отдельный фидбэк-айтем

Каждая отдельная проблема, предложение или вопрос — это один айтем, даже если в исходнике они склеены вместе. Разделяй составные пункты на отдельные айтемы.

### 3. Классифицируй и структурируй каждый айтем

Для каждого айтема заполни следующие поля:

```json
{
  "id": "FB-001",
  "type": "bug | ux_issue | feature_request | question | general_feedback",
  "title": "Краткое, чёткое описание в 5-10 слов",
  "description": "Подробное описание проблемы или предложения. Переписывай сырой текст в понятную формулировку, сохраняя суть",
  "root_cause": "Если указана причина проблемы — кратко здесь, иначе null",
  "area": "Одна из: boards | tasks | documents | navigation | notifications | collaboration | search | settings | performance | mail | access_control | knowledge_base | other",
  "priority": "p1 | p2 | p3 | p4 | unset",
  "complexity": "easy | medium | hard | unset",
  "stage": "Этап/контекст, на котором возникает проблема, если указан",
  "proposed_solution": "Предложенное решение из фидбэка. Если не указано — null",
  "dev_notes": "Технические заметки для разработчика: что именно нужно реализовать, какие модули затрагивает",
  "raw_source": "Оригинальный текст из источника, без изменений (для трассируемости)",
  "status": "new",
  "tags": ["тег1", "тег2"]
}
```

### Правила классификации

**type — тип обращения:**
- `bug` — что-то не работает, ломается, выдаёт ошибку
- `ux_issue` — работает, но неудобно, непонятно, нелогично
- `feature_request` — хочу новую возможность, которой нет
- `question` — непонимание, требуется разъяснение
- `general_feedback` — общее впечатление, оценка, комментарий

**priority — приоритет:**
- `p1` — блокирует работу, критично
- `p2` — сильно мешает, нужно решить скоро
- `p3` — можно обойти, но желательно исправить
- `p4` — мелочь, когда-нибудь
- `unset` — приоритет не указан и не очевиден из контекста

Если в исходнике указан приоритет (p1, p2, "критично", "0" и т.д.) — используй его. Значение "0" в сырых данных обычно означает "не приоритезировано" → `unset`. Не придумывай приоритет если он неясен — лучше `unset`.

**area — область платформы:**
Маппинг из сырых данных: "general" → определи по контексту, "knowledge base" → `knowledge_base`, "mail" → `mail`, "access control" → `access_control`.

**complexity:**
Если указано ("easy", "hard", "medium") — бери как есть. Если нет — `unset`. Не угадывай сложность.

### 4. Добавь dev_notes — самое ценное поле

В `dev_notes` извлеки конкретные требования к реализации:
- Какие UI-компоненты нужны (диалоговое окно, кнопка, колонка)
- Какая логика (алгоритм, триггер, условие)
- Какие интеграции (почта, уведомления, права доступа)
- Связи с другими айтемами (если один перекрывает другой — укажи)

Пример: из сырого текста *"Окно для комментариев если не согласен со сроками... РП может конкретные колонки не возможноми для изменения"* извлеки:
```
"dev_notes": "1) Добавить комментарии к задачам с возможностью прикрепления файлов. 2) Роль РП: возможность блокировки редактирования отдельных колонок. Изменения — через механизм запросов (updates)."
```

### 5. Формат вывода

Выводи результат как JSON-массив. В начале добавь мета-блок:

```json
{
  "meta": {
    "source": "описание источника данных",
    "parsed_at": "дата",
    "total_items": 0,
    "breakdown": {
      "by_type": {"bug": 0, "ux_issue": 0, "feature_request": 0, "question": 0, "general_feedback": 0},
      "by_priority": {"p1": 0, "p2": 0, "p3": 0, "p4": 0, "unset": 0},
      "by_area": {}
    }
  },
  "items": [...]
}
```

### 6. Обработка особых случаев

- **Дублирующиеся пункты**: если один фидбэк перекрывает другой (как в исходнике "перекрывается пунктом по приоритетности") — сохрани оба, но в `tags` добавь `"duplicate_of:FB-XXX"` или `"overlaps:FB-XXX"`
- **Смешанные пункты**: если один отзыв содержит и баг и предложение — разбей на два айтема, в `tags` свяжи через `"related:FB-XXX"`
- **Неполные данные**: заполняй что можешь, остальное `null`. Не придумывай
- **Отмеченные как "всё ок"**: если комментарий означает что решение уже учтено — добавь тег `"acknowledged"`, оставь `status: "new"`

## После вывода JSON

Кратко сообщи пользователю:
- Сколько айтемов извлечено
- Распределение по типам и приоритетам
- Ключевые паттерны (например: "Большинство проблем связано с этапом планирования и нехваткой данных")
- Рекомендации что адресовать первым

Спроси нужно ли сохранить в файл (.json) или нужен другой формат (CSV, таблица для Notion/Jira импорта).
